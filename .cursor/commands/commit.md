# 提交当前代码库改动

在**本仓库根目录**将工作区（与已暂存）的改动整理为 **一次或多次** `git commit`。不执行 `git push`，除非用户在本轮对话中明确要求。

## 执行前

1. 使用终端在仓库根执行并据实分析结果：
   - `git status`
   - `git diff`
   - 若已有暂存： `git diff --staged`
2. 若**无改动**或**仅未合并的纯换行/无意义状态**，向用户说明并结束，不要空提交。

## 安全与范围

- **不要** `git add` 或提交：`.env`、`.env.*`（非 example）、密钥、本机路径、大段误加的二进制/私物。若发现，从暂存中排除并提醒用户。
- 若一次 `git diff` 里混有**两件以上可独立说明的变更**，优先建议**拆成多个 commit**；若用户希望单条提交，在 message 中分点写清，但避免把无关主题硬塞成一句。

## 提交信息

- **第一行**（subject）：用**完整、可独立理解**的一句话概括本次提交；可参照 *conventional commits*（如 `feat: …` / `fix: …` / `chore: …` / `refactor: …`），但**以本仓库已有习惯为准**；英文或中文均可，**全库风格一致**即可。
- 需要时加**空行**后写 body：说明动机、影响面、或拆点；**好语法、不堆缩略语**。

## 操作

1. `git add` 只加入**应进入该次提交**的路径（可用 `git add -p` 在需要时交互分块；由你根据用户意图选择，并说明原因）。
2. 使用 `git commit -m` 多行 message 或 `HEREDOC` 等方式提交，保证 message 与 `git log` 中其他提交**语气与格式**协调。
3. 完成后**简要回报**：本次 commit 的短 hash、subject 行、以及是否还有未暂存/未提交内容（`git status` 一屏总结即可）。

## 用户未说清时

- 仅说「提交」且未给 message 时：你根据 `diff` **起草** subject（+ 必要 body），在执行 `commit` 前**用一句话向用户展示**将使用的首行；若用户曾明确过 message，以用户为准。
